Gestión de Memoria

Iridium optimiza el ciclo de vida de los datos para garantizar que el entrenamiento de modelos de IA y el backend resiliente no agoten los recursos del sistema.

1. Carga Eficiente: Bloque `env`

A diferencia de otros lenguajes que cargan archivos de configuración en memoria dinámica cada vez que se consultan, Iridium utiliza el bloque nativo env para mapear secretos directamente a constantes inmutables al inicio de la ejecución.

env of ".env":
    const API_KEY: string FROM env.file
    const PORT: int = 8080 FROM env.file

2. Ciclo de Vida de Variables (Scopes)

Iridium gestiona la memoria basándose en bloques de indentación. Cuando un bloque (como un if, for o while) termina, el motor libera las variables locales que no fueron marcadas para persistencia.

3. El Operador `Sponge` y la Memoria

El operador ~> no solo ayuda a la resiliencia, sino que evita "Memory Leaks" al asegurar que una variable siempre tenga un valor definido, evitando punteros nulos o estados de error que mantengan buffers abiertos innecesariamente.

4. Futuro: V2 y V3

Actualmente, la V1 de Iridium utiliza el recolector de basura (Garbage Collector) del núcleo de Python. Sin embargo, nuestro Roadmap incluye:

  • V2 (Bytecode): Implementación de una pila de ejecución (Stack Machine) propia.
  • V3 (LLVM): Gestión manual de memoria mediante punteros seguros para máxima eficiencia en IA.

Optimizando Tensores

Las variables de tipo tensor utilizan una asignación de memoria contigua, lo que permite que la CPU o GPU acceda a los datos de entrenamiento con una latencia mínima.
